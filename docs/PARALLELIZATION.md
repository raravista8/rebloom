# PARALLELIZATION — что можно делать параллельно и независимо · Передарим (`rebloom`)

> Карта рабочих потоков: что стартует сразу и идёт параллельно, что от чего зависит, где точки синхронизации. Цель — запустить Claude Design и несколько сессий Claude Code, не блокируя друг друга.

## 1. Что вообще позволяет параллелить — «контракты-развязки»
Потоки независимы, пока стабильны общие контракты:
- **`API_CONTRACT.md`** — развязывает фронт и бек (бек реализует, фронт мокает и потребляет).
- **`@rebloom/canon` + токены** — развязывают дизайн и фронт-код (дизайн отдаёт пакет, фронт вендорит).
- **`DESIGN_BRIEF / SCREEN_INDEX / INTERACTION_STATES / FLOWS / MOTION`** — развязывают дизайн от всего: можно проектировать, зная продукт, без готового бэка.
- **Модель данных (`ARCHITECTURE` ER)** — общая основа для бека и аналитики.
Пока эти документы зафиксированы, потоки не ждут друг друга.

## 2. Рабочие потоки (tracks)
| Track | Кто | Что | Зависит от | Можно начать |
|---|---|---|---|---|
| **D — Design** | Claude Design | дизайн-система, токены, экраны, состояния, флоу, моушн → `@rebloom/canon` | DESIGN_BRIEF/SCREEN_INDEX/INTERACTION_STATES/FLOWS/API_CONTRACT | **сразу** |
| **I — Infra/DevOps** | Claude Code | E0 (скелет+CI), Docker Compose, managed PG/Redis, Object Storage+CDN, Caddy, observability | — | **сразу** |
| **B — Backend** | Claude Code | auth→listings→photos→feed→deals/escrow→delivery→reviews→moderation→notifications | API_CONTRACT + ER + E0 | **сразу после E0** (UI не нужен) |
| **F — Frontend web** | Claude Code | веб-приложение `web/` на canon + API | canon (Track D) + API_CONTRACT (мок) | после первого экспорта canon; **API можно мокать** |
| **M — Mobile** | Claude Code | Capacitor-обёртка, плагины, релиз в сторы | стабильный `web/` build | скелет рано, финал — после F |
| **A — Admin/Analytics/Fraud** | Claude Code + Design | E11 + admin-UI | events/ledger (B), ADMIN_*; admin-canon (D) | параллельно после E1/E5 |

## 3. Полностью параллельно с дня 1
- **Track D (дизайн)** ∥ **Track I (инфра)** ∥ **Track B (бекенд)** — не пересекаются. Дизайн рисует, инфра поднимает CI/окружения, бекенд пишет домен и тестируется **через API-тесты без всякого UI**.
- Внутри бекенда параллелятся независимые модули: `listings/photos/feed/likes`, `reviews`, `notifications`, `moderation`, `analytics` — слабо связаны.

## 4. Что лучше делать последовательно / с осторожностью
- **Денежное ядро `deals`+`payments`+`ledger`** — высокий риск, единые инварианты: вести как один аккуратный поток (TDD, тест-инвариант первым), не дробить между параллельными агентами.
- **Webhook/идемпотентность/релиз** — после базового `deals`.
- **Фронт интеграция с реальным API** — после того как соответствующий бэкенд-эндпоинт готов (до этого — мок по контракту).

## 5. Как фронт не ждёт бек (и наоборот)
- **Фронт → мок API из `API_CONTRACT`**: поднять mock-сервер (формы/enum'ы/ошибки из контракта), разрабатывать экраны и состояния (incl. empty/no-results/error/offline) на моках; переключить на реальный бек по мере готовности эндпоинтов.
- **Фронт → canon как появляется**: пока нет компонента из Claude Design — заглушка с известным API компонента (инвентарь известен из DESIGN_BRIEF), замена при вендоринге.
- **Бек → не нужен UI**: весь домен покрывается unit/integration/contract-тестами; готовность меряется тестами, а не экранами.

## 6. Точки синхронизации (sync points)
1. **Заморозка `API_CONTRACT`** — перед активной разработкой фронта и бека (минимизирует переделки). Изменения контракта → bump + уведомить оба потока.
2. **Первый экспорт `@rebloom/canon`** — фронт переходит с заглушек на реальные компоненты.
3. **Интеграция F↔B на staging** — реальный фронт против реального бека (контрактные/e2e-тесты).
4. **Обёртка M** — после стабильного `web/` build.
5. **Релиз в сторы** — после M + готовых privacy/legal (PRIVACY_152FZ, LEGAL) + ревью-нотов (store-submission).

## 7. Привязка к эпикам TASKS
- **Сразу:** E0 (I), Track D (дизайн), начало E1/E3/E4 (B).
- **Параллельно:** E5 (деньги — отдельный аккуратный поток), E6/E7/E8, E10 (наблюдаемость/деплой — с I).
- **После canon+contract:** E2 (F), затем E9 (M).
- **Параллельно после E1/E5:** E11 (A — admin/analytics/fraud), E12 (уведомления/приватность/поддержка/реферал/пики).

## 8. Рекомендация по запуску
Запускайте **три потока одновременно уже сейчас**: (1) Claude Design = Track D, (2) Claude Code сессия = Track I+B (инфра + бекенд против контракта), (3) как только есть первый canon — Claude Code сессия = Track F (фронт). Деньги (`deals/payments`) — отдельной аккуратной веткой. Admin (E11) и сторы (E9) — позже, не на критическом пути MVP.
