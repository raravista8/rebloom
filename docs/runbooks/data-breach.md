# RUNBOOK — утечка ПДн / инцидент безопасности · Передарим (`rebloom`)

> «План на случай утечки» (ФЗ-152). Цель: за минуты локализовать, в срок уведомить РКН,
> восстановиться без потери данных. Деньги/ledger fail-secure; ПДн — телефоны, имена,
> адреса встреч (AES-256-GCM), IP. Связано: `SECURITY.md` (T-10/T-16, A09), `PRIVACY_152FZ.md`.

## 0. Когда это инцидент
Любое из: несанкционированный доступ к БД/бэкапу/`.env`; утечка дампа/снапшота диска; компрометация
admin-аккаунта или SSH/root; секрет в публичном месте (git/лог/CI); аномалия в `AUDIT_LOG`
(массовый экспорт ПДн, доступ из неожиданного гео/IP); algo-подтверждённый слив (мониторинг,
жалоба, внешнее уведомление). **При сомнении — относимся как к инциденту.**

## 1. Первые 60 минут — ЛОКАЛИЗАЦИЯ (containment)
Веди журнал действий с таймстемпами (UTC) — он нужен для уведомления и пост-мортема.
1. **Заморозить деньги:** ничего не релизим/возвращаем; деньги остаются `paid_held` (fail-secure).
2. **Отозвать сессии:** `redis-cli FLUSHDB` на сессионной БД ⇒ все логин-сессии инвалидируются
   (`docker compose ... exec redis redis-cli -n 0 FLUSHDB`). Пользователи перелогинятся.
3. **Отрезать вектор:**
   - утёк секрет/ключ → §3 ротация;
   - скомпрометирован SSH/box → сменить SSH-ключи, `ufw default deny`, проверить `last`/`/root/.ssh/authorized_keys`, поднять снапшот-реплику в изоляции;
   - утёк бэкап → §3 ротация `BACKUP_ENC_KEY` + проверить, что remote-доступ закрыт;
   - admin-аккаунт → заблокировать (`/api/admin/users/{id}/block`), сбросить его TOTP, проверить `AUDIT_LOG` его действий.
4. **Сохранить улики ДО изменений:** снять копию логов (`docker logs`), `AUDIT_LOG` (append-only),
   текущий снапшот БД — в изолированное хранилище. Не затирать.
5. **Оценить охват:** какие таблицы/поля/строки, сколько субъектов, какие именно ПДн
   (телефон/имя/город/адрес/IP). Адреса встреч зашифрованы AES-256-GCM — если ключ `PII_ENC_KEY`
   НЕ утёк, они остаются нечитаемыми (понизить серьёзность); если утёк — считать раскрытыми.

## 2. Уведомление РКН и субъектов (ФЗ-152, ред. с 2022)
> Сроки `[verify с юристом — действующая редакция]`, ориентир по 152-ФЗ ст.21 ч.3.1:
- **≤ 24 часов** с момента обнаружения — **первичное уведомление в Роскомнадзор** о факте инцидента:
  что произошло, предполагаемая причина, какой вред, какие ПДн и сколько субъектов, какие меры приняты.
  Канал: форма на сайте РКН (реестр инцидентов) / через ответственного за ОПД.
- **≤ 72 часов** — **второе уведомление**: результаты внутреннего расследования, виновные лица (если установлены).
- **Субъектам** — уведомить затронутых пользователей (push/email) без эха конкретных данных,
  с рекомендациями (сменить вход/телефон-привязку, бдительность) — по решению юриста/политики.
- Зафиксировать всё в `AUDIT_LOG` + инцидент-журнале.

## 3. Ротация секретов/ключей (по тому, что затронуто)
Все секреты — в `/opt/rebloom/.env` (root-only). После ротации: `docker compose ... up -d --force-recreate <svc>`.
- **`APP_SECRET_KEY`** (подпись сессий/OTP-HMAC) → ротация инвалидирует все сессии/OTP (нужно §1.2).
- **`PII_ENC_KEY`** (AES-256-GCM адресов) — ⚠️ ротация требует пере-шифрования существующих строк
  (read-old → write-new); см. ADR-0012. Если ключ утёк — приоритет №1, считать адреса раскрытыми.
- **`POSTGRES_PASSWORD`**, **`REDIS_PASSWORD`** → сменить + пересоздать db/redis/api/scheduler.
- **`BACKUP_ENC_KEY`** → сменить; старые бэкапы, зашифрованные старым ключом, считать скомпрометированными если ключ утёк.
- **Провайдеры** (`YOOKASSA_*`, `SMS_API_KEY`, `OAUTH_*`, `SMTP_PASSWORD`, `TG_BOT_TOKEN`) → отозвать в кабинете провайдера, выпустить новые, обновить `.env`.
- **SSH/деплой-ключи** → выпустить новые, удалить старые из `authorized_keys`, проверить CI-секреты (`gh secret`).
- Прогнать `gitleaks` по репо/истории — убедиться, что секрет не в git.

## 4. Восстановление
- Восстановить из **off-box зашифрованного бэкапа** (`infra/scripts/restore.sh` ← `backup.sh`),
  проверить целостность (сверка `AUDIT_LOG`/`LEDGER_ENTRY`), затем вернуть трафик.
- Поднять патчи, закрывающие вектор; прогнать `make security-check` + проверить заголовки/firewall.
- Снять режим инцидента только после подтверждения, что вектор закрыт и данные целы.

## 5. Пост-мортем (в течение недели)
Без поиска виноватых: таймлайн, корневая причина, что сработало/нет, что меняем (контроль, алерт,
тест), статус уведомлений. Завести задачи на устранение. Обновить этот runbook тем, что узнали.

## 6. Профилактика (держать зелёным — иначе инцидент дороже)
Шифрование диска/managed-PG (at-rest), зашифрованные off-box бэкапы по расписанию + тест-restore,
алерты A09 (всплеск failed-login, аномалии экспорта ПДн, провалы webhook-подписи), admin 2FA + IP-allowlist,
SSH key-only/no-root/fail2ban, ретеншн-джобы (чистка IP/сессий), `SMS_REVEAL_OTP` выключен на проде.
