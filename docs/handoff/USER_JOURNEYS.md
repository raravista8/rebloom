# USER JOURNEYS — основные сценарии покупателя и продавца (verified E2E)

> Канонические пути «от знакомства с сайтом и далее». Каждый шаг привязан к экрану
> (`SCREEN_INDEX.md`), API (`API_CONTRACT.md`) и no-escrow-модели (ADR-0013).
> **Проверяется** локальным end-to-end прогоном `web/tests/e2e-local/journeys.mjs`
> против полного стека (реальный backend, без моков) — он гоняет обе роли через OTP и
> доводит сделку до `done`. См. «Как прогнать» внизу.

## Покупатель «Марина»
1. **Знакомство** — заходит на `/` (лендинг): герой, живой каталог (`/api/feed`,
   две топ-ленты «свежие»/«залайканные»), «как это работает», отзывы, «спокойная сделка».
   Мобильный бургер-меню (`PdMobileMenu`), чип города «📍 Москва ▾».
2. **Город** — `NavCity` (поповер на десктопе / `/city` на мобильном).
3. **Каталог** — карточки `BouquetCard` с фильтрами (цена/свежесть/рейтинг); `/search`.
4. **Карточка букета** — `/l/[id]`: фото, свежесть, лайк, цена, продавец+рейтинг.
5. **Лайк** — сердце (идемпотентно, оптимистично).
6. **Связь** — «Написать продавцу» → (гейт входа, если не залогинен) → `POST /deals`
   → сделка `agreed`, открывается `/deal/[id]` с чатом.
7. **Чат** — договариваются о встрече (`DealChat`, contact-leak фильтр).
8. **Встреча** — продавец делится местом → статус `meeting`, адрес раскрыт.
9. **Подтверждение** — «Подтвердить получением» → `done`. Оплата — при встрече (P2P,
   площадка денег не касается).
10. **Отзыв** — `/deal/[id]/review`: оценка + текст (взаимный, 14 дней).

## Продавец «Аня»
1. **Знакомство** — `/`, CTA «Опубликовать букет».
2. **Вход** — `/login`: телефон → OTP (6 цифр) + согласие 152-ФЗ.
3. **Публикация** — `/sell`: 1–5 фото, размер (S/M/L/XL), свежесть (today/1-2д/3+),
   цена, город → модерация → `active` (или `pending_review`).
4. **Покупатель пишет** — приходит сделка `agreed`, открывается чат.
5. **Место встречи** — «Поделиться местом» (`share-point`) → `meeting` (адрес
   шифруется AES-256-GCM, раскрывается покупателю).
6. **Встреча и оплата** — наличными/переводом напрямую, без предоплаты.
7. **Завершение** — покупатель подтверждает → `done`.
8. **Отзыв** — взаимный.

## Состояние реализации (по E2E против реального бэка)
Полный путь покупатель↔продавец (landing → browse → OTP-login → listing → like →
contact → deal → chat → share-point → confirm → review) проходит зелёным, сделка
доходит до `done`. Прогон вскрыл и зафиксировал 3 класса дефектов, которые пустой
прод-каталог маскировал (крашили экран на реальных данных):
- `seller_rating`/`display_name` = `null` у продавца без отзывов/имени — `.toFixed()`
  и аватар `n[0]` падали (лента/карточка/профиль). Гард + фоллбэк «Новый»/«Продавец».
- shape-mismatch: `/deals/{id}`·share-point·confirm·cancel·review возвращают `{deal}`
  (web не разворачивал) и `/messages` возвращает `{messages}` (web читал `.items`).

## Как прогнать локально
```
docker compose -f infra/docker-compose.yml up -d db redis api
docker compose -f infra/docker-compose.yml run --rm api alembic upgrade head   # схема + сид городов
cd web && API_PROXY_TARGET=http://localhost:8000 npm run build && npm run start &
cd web && node tests/e2e-local/journeys.mjs            # OTP читается из логов api (APP_ENV=local)
```
> Это не CI-тест (нужен живой стек + OTP из логов). CI покрывает те же экраны
> мок-спеками (`web/tests/visual/*.spec.ts`) — но моки должны повторять РЕАЛЬНУЮ форму
> ответа бэка (этот прогон поймал, что часть моков повторяла баг web, а не контракт).
