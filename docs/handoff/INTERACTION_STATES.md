# INTERACTION & STATES — ТЗ для Claude Design · Передарим (`rebloom`)

> **Жёсткое требование:** ни один экран и ни один интерактивный компонент не считается готовым, пока не проработаны ВСЕ его состояния — и каждое состояние, где это нужно, **привязано к ответу бекенда** (см. `API_CONTRACT.md`). Это парный документ к `DESIGN_BRIEF.md`.

## 1. Глобальные принципы взаимодействия

- **Кнопки по умолчанию активны.** Rest-состояние любой кнопки — **enabled/active**. `disabled` допускается ТОЛЬКО в двух случаях: (a) действие уже выполняется (in-flight) — чтобы предотвратить двойной сабмит, кнопка переходит в `loading`/`disabled` на время запроса; (b) есть жёсткое объективное предусловие, которое физически нельзя выполнить. **Запрещён паттерн «серая кнопка, пока форма невалидна»** — кнопка остаётся активной, а при нажатии показываем инлайн-валидацию/ошибку. Это касается всех CTA: «Купить», «Опубликовать», «Войти», «Отправить» и т.д.
- **Hover — только для pointer-устройств.** На тач-экранах hover не существует → используем `pressed/active`. Дизайн обязан показать оба: hover (desktop/web) и pressed (touch).
- **Focus всегда видим.** Любой интерактивный элемент имеет видимый `focus-visible` (ring из токена `focus-ring`) для клавиатурной навигации и доступности.
- **Reduced motion.** Все анимации уважают `prefers-reduced-motion`.
- **Никакого «мёртвого» UI.** Любое ожидание → loading/skeleton; любой пустой результат → осмысленное состояние с действием; любая ошибка → понятный текст + повтор.

## 2. Набор состояний по типам компонентов (универсальный)
Для КАЖДОГО компонента из `DESIGN_BRIEF.md §4` проработать применимые состояния:

| Состояние | Кнопки | Карточки/кликабельные | Тогглы/чипы | Примечание |
|---|---|---|---|---|
| `rest` (default = active) | ✓ | ✓ | ✓ | базовое, активное |
| `hover` (pointer) | ✓ | ✓ | ✓ | только desktop/web |
| `focus-visible` | ✓ | ✓ | ✓ | ring, клавиатура |
| `active`/`pressed` | ✓ | ✓ | ✓ | тач + клик |
| `selected`/`on` | — | ✓ | ✓ | для тогглов/табов/лайка |
| `loading` | ✓ | ✓ | — | спиннер/skeleton, блок двойного сабмита |
| `disabled` | редко | редко | редко | только по §1 (a)/(b) |
| `error` | ✓ | ✓ | — | напр. неуспешный сабмит |

## 3. Состояния инпутов (обязательно все)
`rest` → `hover` (pointer) → `focus` (ring) → `filled` (есть значение) → `valid` → `invalid` (+ сообщение об ошибке под полем) → `disabled` → `read-only`. Дополнительно: с иконкой/префиксом, с кнопкой очистки, счётчик символов где есть лимит.
- **OtpInput:** `empty` → `typing` (по ячейкам) → `verifying` (loading) → `invalid` (неверный код, ошибка + повтор) → `locked` (после лимита попыток — текст «слишком много попыток, попробуйте через N мин», ввод заблокирован). Привязка: `otp_locked` из API (FR-003).
- Валидация показывается **по нажатию/блюру**, а не блокирует кнопку заранее (см. §1).

## 4. Жизненный цикл коллекций (лента / поиск / списки) — все состояния
1. `idle/initial` — ещё не запрашивали (напр. пустой поиск без запроса).
2. `loading` — первичная загрузка → **skeleton** карточек (не спиннер на пустом экране).
3. `loaded` — есть данные.
4. `empty` — данных нет в принципе (напр. в этом городе ещё нет объявлений) → дружелюбный empty-state + действие («Будьте первым — опубликуйте букет» / сменить город).
5. `no-results` — запрос/фильтр вернул 0 (поиск «розы» в Самаре — ничего) → **отдельное** состояние: «По запросу „…“ ничего не найдено», + подсказки (сбросить фильтры / другой город / похожие). **Это не то же, что `empty`.**
6. `loading-more` — догрузка следующей страницы (нижний спиннер).
7. `end-of-list` — больше нет (`next_cursor == null`).
8. `error` — запрос упал → текст + кнопка «Повторить».
9. `offline` — нет сети (клиентское) → баннер «Нет соединения», повтор при возврате сети.

## 5. Обязательные состояния по экранам (минимум)
| Экран | Обязательные состояния |
|---|---|
| Главная/витрина | loading(skeleton ленты), loaded, empty(нет объявлений в городе), error, offline |
| Поиск/фильтр | idle(нет запроса), loading, results, **no-results**(+подсказки), error, offline |
| Карточка букета | loading, loaded, `sold/reserved`(недоступно — `listing_unavailable`), error/404 |
| Продать | форма rest, upload-progress(фото), validation-error(инлайн), submitting(loading), success, `moderation_pending`(на модерации) |
| Вход (OTP) | phone rest, sending, otp typing/verifying/invalid/locked, success |
| Сделка | created/paid_held/released/disputed/cancelled, payment_failed(деньги held), loading, error |
| Спор (FLOW-1) | выбор причины, доказательства, ведение(open/на рассмотрении/решён, SLA, empty/sending/error/offline), разрешение саппортом(confirm+reason/in-flight) — детально в `FLOWS.md` |
| Чат сделки | empty(нет сообщений), sending, message-held(контакты вырезаны), error |
| Отзыв | доступен(после released), submitted, held(модерация), уже оставлен |
| Профиль | loading, loaded, empty(нет объявлений/отзывов) |
| Уведомления | loading, list, empty |
| Админ/модерация | loading, queue, empty(очередь пуста), action-success/error |

## 6. Состояния ↔ бекенд (что бек обязан вернуть, чтобы UI различил)
| UI-состояние | Условие/ответ API | Что бек возвращает | Дизайн показывает |
|---|---|---|---|
| `loaded` | 200, есть items | `{items:[…], next_cursor}` | данные |
| `empty` (нет данных вовсе) | 200, items=[], фильтр/запрос НЕ заданы | `{items:[], applied:{q:null,filters:none}}` | empty-state + CTA |
| `no-results` (поиск 0) | 200, items=[], запрос/фильтр заданы | `{items:[], applied:{q,filters}, suggestions?}` | «ничего не найдено» + подсказки |
| `end-of-list` | 200, последняя страница | `next_cursor: null` | стоп-маркер |
| `sold/unavailable` | покупка занятого | `error: listing_unavailable` | «уже продано» |
| `payment_failed` | сбой оплаты | `error: payment_failed`, сделка остаётся `paid_held` | «оплата не прошла», деньги в безопасности |
| `otp_locked` | превышены попытки | `error: otp_locked` (+retry_after) | блок ввода + таймер |
| `forbidden`/`unauthorized` | нет прав/сессии | `error: forbidden`/`unauthorized` | редирект на вход / «нет доступа» |
| `not_found` | нет ресурса | `error: not_found` (404) | 404-экран |
| `rate_limited` | флуд | `error: rate_limited` | «слишком часто, подождите» |
| `moderation_pending` | объявление/отзыв на проверке | `error/flag: moderation_pending` | «на модерации» |
| `content_blocked` | мат/слуры/контакты в тексте | `error: content_blocked` | инлайн-ошибка под полем, БЕЗ эха запрещённого слова (MODERATION.md) |
| `error` (прочее) | 5xx | `error: internal`, без стектрейса | «что-то пошло не так» + повтор |
| `offline` | нет сети | — (клиентское) | баннер offline + retry |

> Где для различения состояний бекенду нужно вернуть доп. данные (эхо `applied` фильтров, `suggestions`, `retry_after`) — это отражено в `API_CONTRACT.md` и в задачах бека (TASKS). **Дизайн не выдумывает статусы — берёт только из контракта.**

## 7. Поиск — детально (раз «может ничего не найтись»)
- `idle`: поле поиска без запроса → показать недавние/популярные подсказки или просто ленту города.
- `loading`: skeleton.
- `no-results`: «По запросу „розы пионовидные“ в Самаре ничего не найдено» + действия: сбросить фильтры, расширить город, показать похожие/свежие. Бек возвращает `applied` (эхо запроса/города/фильтров) и опционально `suggestions`.
- Не путать с `empty` (в городе вообще нет объявлений) — разные тексты и действия.

## 8. Definition of Done (для дизайн-хэндофа)
Экран/компонент готов, только если:
- все применимые состояния из §2–§5 нарисованы (включая hover/focus/pressed, empty, **no-results**, error, offline, loading);
- кнопки активны по умолчанию (§1), disabled — только по правилу;
- состояния, зависящие от данных, **привязаны к условиям `API_CONTRACT.md §7` и §6 этого файла**;
- для каждого состояния есть RU-копирайт и baseline-скриншот для pixel-diff.
