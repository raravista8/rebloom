## ⚠️ 0. Сначала прочесть — три пункта меняют контракт/данные, не только пиксели

Бóльшая часть ниже — чистый canon (визуал/копирайт). **Но эти требуют новых данных + backend-правок**
(иначе экраны покажут контролы без данных). Дизайн рисует UI и **в CHANGELOG помечает заложенные поля**,
бэкенд реализует их в том же контракте — это **отдельная параллельная задача**:

1. **Метро.** Справочник станций по городам (Москва + СПб): станция → линия → **официальный цвет**;
   пересадки = несколько цветных кружков. Поле станции у объявления; станция в ленте/листинге/форме;
   **фильтр по метро = мультивыбор** (несколько станций, семантика «любая из выбранных» = OR). →
   `API_CONTRACT`: `metro` в `listing_card`/`listing`, `metro` в `POST /listings`, `metro[]` в
   `/feed`+`/search`; + сид станций/цветов msk+spb.
2. **Типы цветов.** Фиксированный список (см. §2) → канонический enum/справочник. Поле `flower_types[]`
   у объявления; мультивыбор в форме; **фильтр по типам** (OR). → `API_CONTRACT` + backend.
3. **«Показать N букетов»** в мобильной панели фильтров → нужен **счётчик результатов под текущие
   фильтры** до их применения (либо клиент считает по уже загруженному, либо лёгкий count-эндпоинт). `[verify]`

## 1. Метро (новый ориентир)
- Было: только район («Москва · Патрики»). Стало: **город + станция метро**; в городах с метро станция —
  **главный ориентир**.
- Реальные станции **Москвы и СПб** с **официальными цветами линий**; пересадочные узлы (Киевская,
  Курская, Парк культуры, Комсомольская…) — **несколько цветных кружков**. Города без метро → fallback на район.
- Публикация: текстовое «Район» → **picker с поиском + список станций** (цвета линий).
- **Фильтр по метро** (каталог: сайдбар + мобильная панель; и на лендинге): **мультивыбор** — чекбоксы,
  **теги выбранных станций с крестиком**, логика **«любая из выбранных»**.

## 2. Фильтры (главная + каталог)
- Было: цена, свежесть, рейтинг, размер. Стало: **+ тип цветов** и **+ метро**.
- **Типы цветов (фиксированный список):** Розы · Пионовидные розы · Пионы · Тюльпаны · Гортензия ·
  Хризантемы · Эустома · Ранункулюсы · Альстромерия · Лилии · Полевые. (Мультивыбор, OR.)
- Мобайл: чип **«Фильтры»** раньше ничего не открывал (полный набор был только в десктоп-сайдбаре) →
  теперь раскрывает **inline-панель «Все фильтры»** со всеми группами + **«Сбросить»** / **«Показать N букетов»**.
- Сортировка: на мобиле была длинная лента кнопок (обрезалась, дублировала десктоп) → **дропдаун
  «Сортировка ▾»** на мобиле; **сегментный переключатель** на десктопе.
- Размеры в фильтре подписаны: **S · до 7 · M · 7–15 · L · 15–25 · XL · 25+**.

## 3. Карточки букетов (главная, каталог)
- «L» → размер **подписан и сделан тихим**: приглушённый текст рядом с ценой, **без громкой плашки**.
- Город/район убран → вместо него **станция метро** (город ясен по шапке).
- Рейтинг продавца был у дальнего края (путался с букетом) → **вплотную к имени**.
- Длинное имя продавца ломало строку → **обрезается многоточием**, рейтинг остаётся на месте.

## 4. Бейджи свежести
- «Сегодня / 1–2 дня / 3+ дня» → на карточках **«Свежий»** (вместо «сегодня») + короткие **«1–2 дня»**,
  **«3+ дня»**. В **фильтре свежести** добавлен вариант **«3+ дня»**. (Enum не меняется —
  `today|d1_2|d3_plus` — только подписи.)

## 5. Страница букета
- Шеринг: непонятная иконка → серая текст-кнопка (перетягивала) → **тихая icon-кнопка**.
- **Цена убрана с кнопок**.
- Локация: только район → **город + станция метро**; **самовывоз привязан к станции**.
- Убрать подпись «Самовывоз. Точное место… двор или станцию выбирает продавец».
- Продавец: имя+рейтинг без контекста → блок **«Продавец»** → «Продаёт Аня», подпись **«рейтинг
  продавца»**, **число сделок — отдельной статой** (не путается с рейтингом).
- Размер/состав/локация: мешанина шрифтов, текст наезжал → **единый шрифт/начертание, ровные строки**.
- Состав: серые кликабельные «кнопки» с точками → **просто текст через «·»**.

## 6. Форма добавления букета
- «Район» текстом → **выбор станции метро** (picker с поиском).
- Нет выбора цветов → **необязательный мультивыбор типов цветов** (список §2).
- Убрать хинт «Похожие букеты… 700–1 300 ₽».
- «Свежесть: Сегодня…» → **«Когда букет подарили»**.

## 7. Шапка (десктоп + главная)
- Старый `pdw-nav` (плоский, «Сделки», CTA «Продать букет», несбалансированные иконки) → **актуальная
  шапка как на лендинге**: липкая с блюром, **белые скруглённые контролы**, круглый аватар, **«Избранное»
  (сердце)**, CTA **«Опубликовать букет»**; иконки одного размера.
- ⚠️ Заметка потребителю: app-шапка в `web/` — `WebChrome` (хэндролл, уже сделана auth-aware). При
  вендоринге портируем canon-эталон в `WebChrome`, **сохранив auth-aware-логику** (`useMe()`: гость не
  видит «Избранное»/аватар/«Сделки», видит «Войти»).

## 8. Сделка / профиль / вёрстка
- «Букет M · Патрики» (район сливался) → **«Букет M»**.
- «с 2025 на площадке» → **«10 месяцев на площадке»** (относительный срок от регистрации).
- Десктопные страницы обрезались → **артборды под полную высоту, всё видно**.
- Странный высокий пустой хедер в мобильном вебе → **компактная шапка**.

## 9. Стандарты
- Размеры по российской флористике: **S до 7 · M 7–15 · L 15–25 · XL 25+**; подписи **«стеблей / шт.»**.

  `ListingDetail`, форма `SellForm`, поиск `SearchScreen`, app-шапка `WebChrome`, `BottomNav`. →
  структурные/копирайт-правки §1–§9 **портирую вручную** после вендоринга.
- **импортятся целиком**: SEO-страницы `PdGeoPage`/`PdSafeDeal`/`PdBlog*`, чузер логина. Каталог
  (`catalog-page.md`) — экспортировать **импортируемым** компонентом.

## 12. Приёмка (DoD)
Полный экспорт (`src/`+`dist/`+`dist/canon.css`, build-фиксы `.cjs`, CHANGELOG с пометкой новых
data-полей `metro`/`flower_types` и фиксированного списка цветов) по `CLAUDE_DESIGN_PROMPT §3`;
baseline-скриншоты (mobile 375 + desktop) всех затронутых экранов и состояний (вкл. мобильную панель
«Все фильтры», теги метро, дропдаун сортировки); no-escrow (ADR-0013), без точек в заголовках. После
вендоринга: CSS доезжает, JSX §11-хэндролла портирую руками; `npm run test:visual` ≤ 2%.
**Метро / типы цветов / счётчик «Показать N»** — параллельная backend-задача (§0).
