## ⚠️ 0. Сначала прочесть — два пункта меняют контракт, не только пиксели

Бóльшая часть ниже — чистый canon (визуал/копирайт). **Но два требуют новых данных и backend-правок**
(иначе экраны покажут контролы без данных):

1. **Метро как ориентир.** Нужен: справочник станций по городам (Москва + СПб) со станцией → линия →
   официальный цвет ветки, станции-пересадки = несколько цветов; поле станции у объявления; станция в
   ответе ленты/листинга; станция в фильтре и в форме публикации. → правка `API_CONTRACT` (поле `metro`
   в `listing_card`/`listing`; `metro` в `POST /listings`; фильтр `metro` в `/feed`+`/search`) + сид
   станций/цветов + backend-эпик. **Дизайн рисует UI, но без бэка он нефункционален — это отдельная задача.**
2. **Типы цветов.** Поле `flower_types[]` у объявления; мультивыбор в форме; фильтр по типам. → правка
   `API_CONTRACT` + backend.

Остальное (копирайт свежести, «Размер L», шапка, шеринг, вёрстка, размеры-стандарты) — canon-only,
доезжает пакетом. **Дизайн: пометь в CHANGELOG, какие поля данных он закладывает (`metro`,
`flower_types`), чтобы бэкенд их реализовал в том же контракте.**

## 1. Метро (новый ориентир)
- Было: только район («Москва · Патрики»). Стало: **город + станция метро** с цветными кружками линий;
  в городах с метро станция — **главный ориентир**.
- Реальные данные **Москвы и Санкт-Петербурга** с **официальными цветами веток**; станции-пересадки
  (Киевская, Курская, Парк культуры, Комсомольская…) показывают **несколько цветов**.
- Города без метро — fallback на район/город (предусмотреть состояние «без метро»).

## 2. Карточки букетов (главная, каталог)
- «L» → **«Размер L»**.
- Город убран с карточки (он понятен по шапке) → вместо него **станция метро**.
- Рейтинг продавца: был прижат к краю (путался с букетом) → **вплотную к имени продавца**.

## 3. Теги свежести
- «Сегодня» / «1–2 дня» / «3+ дня» → **«Подарен сегодня»** / **«Подарен 1–2 дня назад»** /
  **«Подарен 3+ дня назад»**. (Enum не меняется — `today|d1_2|d3_plus` — меняется только подпись.)

## 4. Страница букета
- Шеринг: непонятная иконка → серая кнопка с текстом (перетягивала) → **тихая понятная icon-кнопка «Поделиться»**.
- **Цена убрана с кнопок** (не дублируется).
- Локация: только район → **город + станция метро**.
- Убрать подпись «Самовывоз. Точное место появится в чате… двор или станцию выбирает продавец» →
  **«Как забрать» = «Самовывоз у м. …»**.
- Продавец: имя+рейтинг без контекста → блок **«Продавец»** + «Продаёт Аня», рейтинг подписан
  **«рейтинг продавца»**, **число сделок — отдельной статой** (не путается с рейтингом).
- Размер/состав/локация: каша шрифтов, текст наезжал → **единый шрифт/начертание, ровные строки**.
- Состав: серые кликабельные «кнопки» с точками → **просто текст через «·»**.

## 5. Форма добавления букета
- «Район» текстом → **выбор станции метро** (поиск + список с цветами линий).
- Нет выбора цветов → **необязательный мультивыбор типов цветов**.
- Убрать хинт «Похожие букеты… 700–1 300 ₽».
- «Свежесть: Сегодня…» → **«Когда букет подарили»**.

## 6. Фильтры (главная + каталог)
- Было: цена, свежесть, рейтинг, размер. Стало: **+ фильтр по станции метро** и **+ фильтр по типам цветов**.
- Размеры подписаны: **S · до 7 · M · 7–15 · L · 15–25 · XL · 25+**.

## 7. Шапка (десктоп + главная)
- Старый `pdw-nav` (плоский, «Сделки», CTA «Продать букет», несбалансированные иконки) → **актуальная
  шапка как на лендинге**: липкая с блюром, белые пилюли, круглый аватар, **«Избранное» (сердце)**,
  CTA **«Опубликовать букет»**; иконки одного размера.
- ⚠️ Потребительская заметка: app-шапка в `web/` — это `WebChrome` (хэндролл, и я только что сделал её
  auth-aware: гость → «Войти»). Дизайн отдаёт **canon-эталон шапки**; при вендоринге шапку портируем в
  `WebChrome`, сохранив auth-aware-логику (`useMe()`: гость не видит «Избранное»/аватар/«Сделки»).

## 8. Сделка / профиль / вёрстка
- Заголовок «Букет M · Патрики» (район сливался) → **«Букет M»**.
- «с 2025 на площадке» → **«10 месяцев на площадке»** (относительный срок от регистрации).
- Десктопные страницы обрезались → **артборды под полную высоту, всё видно**.
- Странный высокий пустой хедер в мобильном вебе → **компактная шапка** (убран notch-отступ).

## 9. Стандарты
- Размеры по российской флористике: **S до 7 · M 7–15 · L 15–25 · XL 25+**; подписи **«стеблей»/«шт.»**.

## 11. Потребительские поверхности — что хэндроллится в `web/` (важно для приёмки)
CSS доезжает пакетом всегда; **JSX этих экранов — нет** (порт руками, см. `CLAUDE_DESIGN_PROMPT §4`):
- **хэндролл** (борют canon CSS, JSX свой): лендинг `LandingHome`, карточка `BouquetCard`, листинг
  `ListingDetail`, форма `SellForm`, поиск `SearchScreen`, app-шапка `WebChrome`, мобильный
  `BottomNav`. → структурные/копирайт-правки §2–§9 **портирую вручную** после вендоринга.
- **импортятся целиком** (JSX доедет сам): SEO-страницы `PdGeoPage`/`PdSafeDeal`/`PdBlog*`,
  чузер логина. Каталог (`catalog-page.md`) — экспортировать **импортируемым** компонентом.

## 12. Приёмка (DoD)
Полный экспорт (`src/`+`dist/`+`dist/canon.css`, build-фиксы `.cjs`, CHANGELOG с пометкой новых
data-полей `metro`/`flower_types`) по `CLAUDE_DESIGN_PROMPT §3`; baseline-скриншоты (mobile 375 +
desktop) всех затронутых экранов и состояний; no-escrow (ADR-0013), без точек в заголовках. После
вендоринга: CSS-правки доезжают, JSX §11-хэндролла портирую руками; `npm run test:visual` ≤ 2%.
**Метро/типы-цветов** — параллельная backend-задача (см. §0), без неё контролы рисуем, но данных нет.
