## 0. Рамки + ⚠️ что НЕ только дизайн (параллельные backend/контракт-задачи)

### 0.1 Что меняем (4 вектора + 2 правки)
1. **Позиционирование → селфи/контент-лид** (вместо «скидка + передаривание»).
2. **Только Москва** (убрать мультигород; SEO пересобрать на районы/метро/повод/тип).
3. **PWA** (вместо нативных приложений и Telegram-бота как основного канала).
4. **Проверка букета кодом** (proof-of-possession) → бейдж «Проверен».
5. **Экономика:** убрать комиссию, публикация бесплатна на старте.
6. **Новый компонент:** пост-самовывозный «Передать букет дальше» — ядро циркуляции.

### 0.2 Анти-скоуп (НЕ достраивать)
Нет эскроу/холдирования (оплата при встрече — ADR-0013, остаётся). Нет полного аукциона (фикс-цена +
«Предложить цену»). **Нет нативных приложений** (iOS/Android/RuStore — только PWA). **Нет других городов**
(никаких переключателей, гео-страниц не-Москвы, «искать во всех городах»). Нет встроенной доставки (только
самовывоз).

### 0.3 Ключевой принцип (легко нарушить)
**Подача (demand) сужается до селфи/контента. Приём предложения (supply) остаётся широким.** Витрина и
маркетинг говорят покупателю «роскошный букет для контента за копейки». Но форма публикации принимает **любой
свежий букет** (передаренный, остатки салона, после фотосессии). **Не делать форму публикации «только для селфи».**

### 0.4 ⚠️ Что НЕ только дизайн — параллельные backend/контракт-задачи
Дизайн рисует UI и помечает заложенные поля; бэкенд реализует в `API_CONTRACT`. Эти пункты — **не чистый canon**:
1. **Проверка кодом (proof-of-possession):** бэкенд выдаёт кодовое слово, принимает фото/видео-подтверждение,
   модерация (`core/moderation`), поле `verified`+`verified_at` у листинга, бейдж в `listing_card`/`listing`. → новый флоу + поля + API.
2. **Счётчик циркуляции:** поле `circulation_count` у листинга, **наследуется при relist** (+1). → поле + relist-логика + в `listing_card`.
3. **«Передать дальше» (relist):** из завершённой сделки создать предзаполненный листинг (фото/тип/размер
   наследуются, свежесть +1 ступень, район из профиля), догоняющий пуш через ~1 день. → `core/deals`+`listings`+`notifications`.
4. **Москва-only:** убрать мультигород из данных/роутинга; **SEO-рестрактур** (`/[район]`, `/metro/[станция]`,
   `/povod/[повод]`, `/[тип]`, комбинации) — крупная routing+sitemap+SSG-задача, не canon-копия. → `geo`/`feed`/web-routing/`sitemap`.
5. **Контент-поводы** (таксономия поводов: «для фото», «для сторис/контента», «себе для настроения») + **фильтр
   «Проверенные»**. → listing-таксономия + `/api/search` фильтры.
6. **PWA install + web-push:** дизайн рисует install-prompt + push pre-prompt (§3.1/3.2); **реализация — `pwa-launch-plan.md`** (manifest/SW/`WebPushProvider`).
7. **Экономика:** убрать «комиссия» из всех полей/копий; публикация бесплатна; **скрытые** хуки монетизации
   (флэт-фи за публикацию, платное продвижение) — спроектировать места, **не активировать**.
   *(Согласуется с ADR-0013 — площадка денег не касается; canon-копию «комиссия 5%» мы уже НЕ портировали.)*

**Чисто canon (визуал/копирайт, доезжает пакетом):** репозиционирование/тон, тексты был→станет, карточка-бейджи
(визуал), секции лендинга, soft-gate/install/pre-prompt UI (компоненты), все состояния.

## 1. Стратегические решения (глобально, на все экраны)

### 1.1 Позиционирование: селфи/контент-лид
**Убрать как ведущее:** «подарили, не нужен, вместо мусорки, вторая жизнь, неловко продавать подарок».
**Ведущее (новое):** аспирационное, контент-нативное, петля «купил → сфоткался → передал дальше».
- Покупателю: *роскошный букет для фотосессии/сторис/настроения, рядом, за десятую часть цены салона.*
- Петля: *наснимал — букет ещё свежий — передай дальше и верни деньги.*
- «Передаривание» — **один из** источников supply, не идентичность бренда.
**Тон:** лёгкий, желанный. **Стоп-слова в заголовках/CTA:** «мусорка», «стыдно/неловко», «секонд-хенд»,
«ненужный», «выбросить» (допустимы мелким планом внутри, не в hero/CTA).

### 1.2 Только Москва
Убрать переключатель города из всех навигаций (хедер/таб-бар/drawer) → статичная подпись «Москва». Убрать чипы
городов и секцию «в других городах», «искать во всех городах», «смените город». Гранулярность переносится
**внутрь Москвы**: районы + станции метро (усилить существующее).

### 1.3 PWA вместо нативки и Telegram
Убрать store-бейджи (App Store/Google Play/RuStore/AppGallery) и «загрузите/установите/откройте в …». Секция
«Приложение» → «Установите на телефон + включите уведомления» (§3.1/3.2). Каналы: **web-push (основной) + SMS
(транзакционный) + email**; Telegram убрать из основных (в настройках — «может работать с перебоями»). **SMS
обязателен для критичного** (OTP уже есть; статус сделки, новое сообщение) — web-push на iOS слабый и только
для установленной PWA.

### 1.4 Проверка букета кодом (proof-of-possession)
Шаг в публикации (§4.2): продавец подтверждает, что букет у него — фото с бумажкой, где написано выданное
**кодовое слово** (напр. «батон»), **или** короткое видео с жестом/словом. После прохождения — бейдж
**«Проверен»**. Доверие без эскроу. На старте обязательно, но быстро (одно фото).

### 1.5 Экономика
Убрать «комиссия 5%» отовсюду. Единая формулировка: **«Публикация бесплатна. Сервис не берёт комиссию со
сделки и не хранит ваши деньги — оплата проходит напрямую при встрече.»** Заложить (но **скрыть**) места под
флэт-фи и платное продвижение.

### 1.6 Цены и торг
Фикс-цена — дефолт. «Предложить цену» — лёгкий торг (без аукциона). Кнопки карточки: «Предложить цену» +
«Написать продавцу».

## 2. Модель авторизации (auth-матрица)
Принцип: **максимально открыть просмотр** (SEO + привлечение), закрывать только действия. После логина —
**вернуть к исходному действию** (deep-link назад).

| Экран / действие | Гость | Авторизован |
|---|---|---|
| Лента/каталог/карточка/профиль (чтение) | ✅ полный | ✅ |
| Лайк/избранное | 🔒 soft-gate (§3.3) | ✅ |
| Написать продавцу / предложить цену | 🔒 soft-gate | ✅ |
| Опубликовать букет | 🔒 soft-gate | ✅ |
| Сделки и чат | 🔒 (нет вкладки → «Войти») | ✅ |
| Передать дальше (relist) | 🔒 soft-gate | ✅ |
| Подписка на push по району | ✅ (анонимно, по устройству) | ✅ (к аккаунту) |
| «Пожаловаться» (чужой профиль/лот) | 🔒 (или с антиспамом, логировать) | ✅ |
| Настройки/уведомления/приватность | — нет доступа | ✅ |

*(Согласуется с нашим `useMe()`-гость-хедером: гость видит «Войти», без избранного/аватара/сделок.)*

## 3. Сквозные паттерны (определить один раз)

### 3.1 PWA-установка (install-prompt)
- **Android/Chrome:** ловить `beforeinstallprompt` → кастомная плашка «Установить приложение» (не сразу — после
  ценностного действия / на 2-м визите) → нативный промпт.
- **iOS/Safari:** нативного промпта нет → карточка-инструкция «Поделиться → На экран „Домой"» (иконки шагов).
- **Установлено** (`display-mode: standalone`) — плашку скрыть. Состояния: доступно / iOS-инструкция /
  установлено(скрыто) / неподдерживается(скрыто).

### 3.2 Web-push прайминг (pre-prompt → системный промпт)
Никогда не вызывать браузерный запрос «в лоб».
- **Триггер pre-prompt:** ценностный момент (после первого лайка / N карточек / явный интент «уведомить о свежих»).
- **Pre-prompt:** «Свежий букет рядом живёт считанные часы. Прислать пуш, когда в {районе} появится подходящий?»
  — «Включить» / «Не сейчас». Только после «Включить» — системный запрос.
- **Гость:** подписка анонимная, по устройству, с выбором района/метро + фильтров. **Авторизован:** к аккаунту.
- Состояния: не спрашивали / pre-prompt / разрешено / отклонено («включить в настройках браузера», не долбить) /
  не поддерживается.

### 3.3 Контекстный логин-волл (soft-gate)
При закрытом действии — не общий блок, а контекстный лист/модалка с причиной + быстрым входом: «Войдите, чтобы
написать продавцу / выставить букет / сохранить в избранное / передать дальше». Внутри — провайдеры из §5.8.
После входа — возврат к действию.

### 3.4 Карточка букета (переиспользуемый компонент)
Единый для лент/каталога/поиска/профиля/«похожих». Состав: обложка, цена, тип/размер, район + метро, бейджи.
**Бейджи:** свежесть («Свежий»/«1–2 дня»/«3+ дня»), размер (S/M/L/XL), **«Проверен»** (новый), **«Передаривали
N раз»** (новый, §4.3; если 0 — скрыть). Состояния: дефолт / в избранном / продано / снято (в «моих») / скелетон.

### 3.5 Состояния коллекции (единый набор)
Загрузка (скелетоны) · догрузка («Показать ещё») · конец («Показали все букеты по вашему запросу») · пусто (нет
данных — заголовок+текст+CTA, **Москва-only**) · ничего-не-найдено (фильтры: «Ничего не нашлось» + «смягчите
цену/свежесть» + «Сбросить фильтры», **без «другие города»**) · ошибка («Повторить») · нет-сети (PWA-fallback).

## 4. Новые компоненты и флоу (ядро редизайна)

### 4.1 «Передать букет дальше» (флайвил циркуляции) — КРИТИЧНО ⚠️backend
**Где:** экран завершённой сделки `/deal/[id]` (статус «завершено»), для **покупателя**, рядом с отзывом.
**UX:** блок «Сфоткались? Букет ещё свежий — передайте его дальше за минуту и верните часть денег.» → кнопка
**«Передать этот букет дальше»** → форма публикации (§5.7) **предзаполненная** (фото/тип/размер наследуются,
свежесть авто +1 ступень, район из профиля); меняет только цену/свежесть, при необходимости — переснять +
проверка кодом заново. Вторичная: «Оценить продавца». **Догоняющий пуш (~1 день):** «Ваш букет ещё свежий?
Передайте дальше» → диплинк в предзаполненную публикацию (уважать настройки каналов).
**Состояния:** доступно (после подтверждения получения) / уже передан («Вы передали букет — спасибо, что
продлили ему жизнь» + ссылка на новый лот) / истёк срок свежести (мягко).

### 4.2 Проверка букета кодом ⚠️backend
**Где:** шаг в `/sell` после фото, до публикации. Система выдаёт **кодовое слово**. Два способа: (а) фото букета
с бумажкой, где видно слово; (б) видео-кружок (показать букет + слово). Подсказка «Это защищает покупателей.
20 секунд.» Состояния: код выдан / загрузка / на проверке / **проверен** (→ бейдж) / не прошло («Не разглядели —
переснимите»). На карточке/в профиле «Проверен» = маркер доверия вместо эскроу.

### 4.3 Счётчик циркуляции ⚠️backend
Поле `circulation_count` (наследуется при relist §4.1). На карточке «Передаривали N раз» + крупнее на странице
букета как мини-история пути (аспирационный соц-пруф). Если 0 — скрыть.

### 4.4 Контент-нативные хуки ⚠️backend(таксономия)
В поводы/фильтры/SEO добавить «Букет для фото», «для сторис/контента», «себе для настроения» — ведущие поводы под
новое позиционирование, рядом с существующими (свидание, ДР, в подарок, себе домой).

## 5. Экраны и состояния (маппинг на наши роуты/компоненты)

> Для каждого: назначение, гость/авторизован, состояния, направление копирайта (был→станет). **«Наш компонент»** —
> что правится у нас; **порт** — хэндролл (JSX руками) или canon-импорт (см. §9).

### 5.1 Лендинг `/` (гость) — наш `components/marketing/LandingHome.tsx` (**хэндролл**)
Секции: Hero → Живой каталог → Как работает (петля) → Циркуляция/истории → Доверие (проверка кодом + отзывы) →
Возражения → Установка+уведомления (PWA) → Финальный CTA (split) → Футер.
- **Hero:** кикер «Роскошные букеты для твоего контента»; H1 **«Шикарный букет для фото — рядом и в разы дешевле
  салона»**; лид **«Возьмите роскошный букет для фотосессии, сторис или настроения за полцены. Наснимали —
  передайте дальше и верните деньги.»**; кнопки «Смотреть букеты» (перв.) · «Передать свой букет» (вторич.);
  трасты «В разы дешевле салона · Оплата при встрече · Проверенные букеты»; бейдж «{N} свежих букетов рядом».
- **Живой каталог:** Москва-only «{N} свежих букетов в Москве сейчас», фильтры как в каталоге, без чипов городов.
- **Как работает (3 шага в круг):** 1 «Нашли букет рядом» · 2 «Забрали и сняли контент» · 3 «Передали дальше —
  круг замыкается»; подзаголовок: покупатель и продавец — часто один человек.
- **Циркуляция/истории:** путь букета (передаривали N раз) + мини-истории (контент/настроение); отзывы без стигмы.
- **Доверие:** проверка кодом + оплата при встрече + рейтинг/отзывы.
- **Возражения:** «подвявший?» (свежесть+бейдж) · «не придёт?» (оплата при встрече + проверка кодом) ·
  «перепродавать — нормально?» (аспирационно: даёте красивому вторую жизнь и контент другим).
- **Установка+уведомления** (бывшая «Приложение»): H2 «Поставьте на телефон и узнавайте о свежих первыми»; убрать
  store-бейджи; install-prompt (§3.1) + push pre-prompt (§3.2).
- **Финальный CTA (split):** покупатель «Роскошный букет за полцены рядом» → «Смотреть букеты в Москве»; держатель
  «Передайте его дальше за минуту» → «Публикация бесплатна, комиссии со сделки нет» · «Передать букет».
- **Футер:** убрать «магазин дешевле» как идентичность; Москва-only; убрать другие города; оставить юр-блок (152-ФЗ).

### 5.2 Каталог `/catalog` (гость+авторизован) — `CatalogScreen.tsx` → canon **`PdCatalog`** (импорт, renderCard=BouquetCard)
H1 «Свежие букеты в Москве», крошки без переключателя города. Фильтры: Метро, Цена, Свежесть, Тип цветов, Рейтинг
продавца, Размер + **«Проверенные»** + контент-поводы (§4.4). Сортировка: свежие/дешевле/дороже/рейтинг. Гость:
лайк/«написать» на карточке → soft-gate. Состояния §3.5; пусто «В Москве сейчас нет свежих букетов. Загляните
позже — их добавляют каждый день» + «Передать свой букет».

### 5.3 SEO-слой — пересобрать под Москву ⚠️backend/routing — canon `PdGeoPage`/`PdSafeDeal`/`PdBlog*` (**импорт целиком**) + наш web-routing
**Новые URL (внутри Москвы):** `/` (лендинг) · `/[район]` · `/metro/[станция]` · `/povod/[повод]` · `/[тип]` +
комбинации `/[район]/[тип]`. **Удалить:** `/[city]` не-Москвы, «в других городах», «работает в 10 городах».
**Шаблон района/метро** (адаптировать город→район): H1 «Свежие букеты {в районе / у м. X}, в разы дешевле салона»;
интро (контент+петля+полцены, Москва-only); секции: живой каталог района → соседние районы → под повод
(контент-поводы) → FAQ → внутренние ссылки на типы/поводы. **FAQ** без «постоял день, не нужен» как единственного
объяснения; + контент-мотив + «букеты проверены»; «как забрать» — самовывоз у метро (доставки нет). **`/bezopasnaya-sdelka`:**
оставить (оплата при встрече), усилить блоком проверки кодом, убрать намёки на эскроу, единая «нет комиссии».
**Блог:** оставить + материалы под новое позиционирование. **Мета карточки:** Title `{Цветы} за {цена} ₽ —

## 9. Наши поверхности: хэндролл vs импорт (порт-карта для вендоринга)
CSS доезжает пакетом всегда; **JSX этих экранов — нет** (порт руками после вендоринга, `CLAUDE_DESIGN_PROMPT §4`):
- **Хэндролл** (борют canon-CSS, JSX наш): `LandingHome`, `BouquetCard`, `ListingDetail`, `SellForm`, `SearchScreen`,
  `WebChrome`(шапка), `HomeFeed`, фильтры каталога/поиска. → структурные/копирайт-правки §1–§8 портирую вручную.
- **Импорт целиком** (JSX доедет сам): SEO-шеллы `PdGeoPage`/`PdSafeDeal`/`PdBlog*`, чузер логина, **`PdCatalog`**
  (его карточку мы отдаём через `renderCard={BouquetCard}` — настоящие фото+лайки; учесть при правках карточки).
- **Новые компоненты** (install-prompt, push-pre-prompt, soft-gate-лист, «Передать дальше»-блок, шаг проверки кодом,
  бейдж «Проверен», счётчик циркуляции) — спроектировать в canon как **data-driven** (пропсы/слоты), не демо: фото —
  **полный URL** (`photoUrl`), действия — наружу (`onLike`/`onSubmit`/`renderCard`); иначе web не сможет импортить
  (грабли 0.9.x — `PdCard`/`PdCatalog`).
