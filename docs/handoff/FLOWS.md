# FLOWS — детальная проработка сложных флоу · Передарим (`rebloom`)

> Пошаговая раскладка ветвящихся сценариев для Claude Design: каждый флоу = entry points → шаги/экраны/модалки → состояния (по `INTERACTION_STATES.md`) → действия → вызовы API (`API_CONTRACT.md`) → копирайт → крайние случаи → уведомления. Дополняет `SCREEN_INDEX.md` (экраны) и `SUPPORT.md` (политика). Каждый шаг рисуется со всеми состояниями (loading/empty/error/success), кнопки активны по умолчанию.

---

## FLOW-1 — Спор по сделке (dispute) — **детально**

### Точки входа
- Экран сделки, статус `paid_held` (и после передачи/доставки до релиза): у покупателя — кнопка «Проблема со сделкой», рядом с «Подтвердить получение»; у продавца — «Открыть спор».
- Из консоли саппорта спор может быть открыт оператором (по обращению).

### Кто и когда
- До `released`/возврата любая сторона может открыть спор → сделка `disputed`, **деньги заморожены** (fail-secure). После релиза — только обращение в поддержку (не меняет деньги автоматически).

### Шаг 1 — Выбор причины (экран «Открыть спор»)
- Покупатель (radio): не получил букет · не соответствует (несвежий/вялый/не тот) · повреждён · продавец не выходит на связь · другое.
- Продавец (radio): покупатель не забирает · не подтверждает получение после передачи · другое.
- Состояния: rest, выбран, error (не выбрано → инлайн), next активна по умолчанию.

### Шаг 2 — Описание + доказательства
- Поле описания (RU, лимит, счётчик) + загрузка 1–5 фото (`PhotoUploader`: upload-progress, ошибка, превью). Текст проходит модерацию контактов/мата (`content_blocked` инлайн).
- Действие «Отправить спор» → `POST /api/deals/{id}/dispute {reason, photo_ids[]}` → `disputed`.
- Состояния: submitting (loading), success (переход в ведение спора), error (повтор).

### Шаг 3 — Ведение спора (экран «Спор»)
- **Шапка:** статус (`открыт` → `на рассмотрении` → `решён`), **SLA-таймер** («ответ поддержки в течение 24ч»), сумма заморожена.
- **Таймлайн/переписка:** сообщения покупатель ↔ продавец ↔ поддержка (`ChatBubble`, system-сообщения о действиях), отправка сообщения + доказательств (`POST /deals/{id}/messages`, модерация контактов).
- **Доказательства:** галерея фото обеих сторон.
- **Действия стороны:** добавить сообщение/фото; принять предложенное решение (если поддержка предлагает); отозвать спор (если договорились) → возврат к `paid_held`.
- Состояния: loading, empty (пока нет сообщений), sending, message-held (вырезаны контакты), error, offline; resolved (read-only итог); sla-overdue (визуальный акцент).

### Шаг 4 — Разрешение (сторона саппорта, в админке) — экран «Кейс спора»
- Контекст: факты сделки, обе стороны + рейтинги, доказательства, чат, ledger/платёж, способ доставки/трекинг.
- **Действия (с обязательной причиной → audit):** релиз продавцу · полный возврат покупателю · **частичный возврат** (ввод суммы, валидация ≤ сумма сделки) · запросить доп. инфо (вернуть в `на рассмотрении`) · эскалация support→admin. **4-eyes** для денежных решений выше порога.
- Состояния: loading, confirm-destructive (модалка + причина), in-flight, success/error.

### Шаг 5 — Итог
- Сделка → `released` / `refunded` / частичный (ledger проводки + чек/возврат 54-ФЗ, fail-secure). Обе стороны видят итог + уведомление (push/email/TG). Отзывы: при возврате — правила оператора (по умолчанию отзыв возможен, но помечен «спор»).

### Крайние случаи
- **Тайм-аут стороны:** нет ответа N дней → авто-решение по политике (напр. при не-подтверждении получения после доставки-подтверждённой → релиз; при не-передаче → возврат) — задаётся политикой `[verify]`.
- **Оба открыли спор** одновременно → один кейс.
- **Чарджбэк** параллельно → флаг, заморозка, ручной разбор.
- **Самовывоз vs курьер:** для курьера учитывается трекинг/подтверждение доставки как доказательство.
- **Скоропортящийся товар:** окно спора короткое (цветы) — подчеркнуть в копирайте сроки.

### API/состояния (привязка)
`POST /deals/{id}/dispute`, `GET /deals/{id}` (включает dispute), `GET/POST /deals/{id}/messages`, admin `POST /admin/deals/{id}` (resolve {action, amount?, reason}). Ошибки/состояния — `INTERACTION_STATES §6`.

## FLOW-2 — Возврат / частичный возврат
Триггер: решение спора/саппорта. Шаги: подтверждение (сумма, причина) → in-flight → ledger-проводка возврата + сторно-чек (54-ФЗ через ЮKassa) → статусы `refunded`/частичный → уведомления. Состояния: confirm, processing, success, error (деньги остаются held при сбое — fail-secure). UX покупателя: «Возврат N₽ отправлен, поступит в срок банка».

## FLOW-3 — Отмена сделки
Варианты: (a) до оплаты — авто-отмена по тайм-ауту 30 мин (`cancelled`, листинг освобождается); (b) покупатель отменяет неоплаченную; (c) саппорт отменяет `paid_held` (= возврат, FLOW-2). Экраны: подтверждение отмены (причина для саппорта), освобождение листинга, уведомления.

## FLOW-4 — Оплата: ретрай и сбой
`payment_failed` → экран «Оплата не прошла» (деньги в безопасности, ничего не списано/средства held), действия: повторить, сменить метод (карта/СБП). Тайм-аут оплаты 30 мин → отмена резерва. Состояния: pending(ожидание webhook), failed, retry, success. Никогда не релизим при неоднозначном ответе провайдера.

## FLOW-5 — Модерация отклонила объявление + апелляция
Публикация → `pending_review`/`content_blocked`. Если отклонено: экран «Объявление не прошло проверку» с общей причиной (без эха запрещённого слова), действия: отредактировать и переотправить · «Не согласен» (апелляция → очередь модерации, `appealed`). Состояния: pending («на проверке ~N мин»), rejected, appealed, approved.

## FLOW-6 — Доставка курьером (трекинг) и фоллбэк
Выбор «курьер» → создание заявки (Яндекс Доставка) → статусы трекинга в сделке. Сбой создания заявки (`delivery claim failed`) → сделка остаётся `paid_held`, экран-выбор: перейти на самовывоз или отменить (FR-032). Адрес раскрывается только после `paid_held` (T-13).

## FLOW-7 — Онбординг самозанятости (опционально, по порогу)
При превышении порога выплат продавцу — необвязывающий промпт «Оформить самозанятость для чеков/налогов» → шаги привязки (через провайдера/ФНС) → статус в профиле. Не блокирует обычные продажи (ADR-0005). Состояния: prompt, in-progress, linked, skipped.

## FLOW-8 — Блокировка/ограничение аккаунта
Заблокированный/ограниченный юзер: экран-заглушка с причиной (обобщённо) + действие «Обжаловать» (тикет в поддержку). Ограниченный (`limited`) — часть действий недоступна с пояснением. Связано с антифродом/модерацией.

## FLOW-9 — Удаление аккаунта (DSR, ФЗ-152)
Настройки → «Удалить аккаунт» → экран-предупреждение (что удалится/что сохранится по закону) → подтверждение (повторный OTP) → soft-disable немедленно + расписание удаления/обезличивания (PRIVACY_152FZ). Состояния: confirm, scheduled, error.

## FLOW-10 — Жалоба на объявление/пользователя
Кнопка «Пожаловаться» на карточке/профиле → выбор причины + комментарий → `POST /api/reports` → благодарность. Уходит в очередь модерации (FR-064). Состояния: form, submitting, success.

## DoD флоу (для Claude Design)
Каждый флоу: все шаги нарисованы как экраны/модалки со **всеми** состояниями (`INTERACTION_STATES`), включая ошибки/тайм-ауты/крайние случаи из этого файла; действия привязаны к `API_CONTRACT`; есть RU-копирайт и baseline-скриншоты. Диспут (FLOW-1) — проработать максимально детально (сторона пользователя и сторона саппорта).
