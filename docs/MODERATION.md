# MODERATION — авто-модерация контента · Передарим (`rebloom`)

> Политика и механика автоматической модерации UGC (объявления, чат, отзывы, имена). Применяется в `core/moderation`. Связано с `SECURITY.md` (T-05/T-09/T-15), `API_CONTRACT.md` (`content_blocked`), `INTERACTION_STATES.md` (инлайн-ошибка ввода).

## 1. Что модерируем и где
| Поверхность | Текст | Фото |
|---|---|---|
| Публикация объявления (заголовок/описание) | ✓ | ✓ |
| Сообщения в чате сделки | ✓ | — |
| Отзывы | ✓ | — |
| Имя профиля | ✓ | — |

## 2. Что блокируем (текст)
- **Мат / обсценная лексика** (RU + транслит).
- **Язык вражды и оскорбления по этническому/национальному/религиозному/иному признаку** — в т.ч. этнические дерогативы (например, унизительные клички по национальности вроде «хохл*» и аналогичные); призывы к ненависти/насилию.
- **Контакты для увода сделки в офлайн** — телефоны, @никнеймы, ссылки на мессенджеры (см. SECURITY T-05).
- **Запрещённое к продаже / мошеннические паттерны**, спам, ссылки на сторонние площадки.

## 3. Как блокируем — пайплайн (fail-closed)
1. **Нормализация ввода** перед проверкой (чтобы ловить обходы): приведение к нижнему регистру; снятие диакритики; замена leet/гомоглифов (`0→о`, `3→е`, латиница↔кириллица), удаление повторов/разделителей внутри слова (`х о х л`, `х*хл`, `хоххл`), unicode-нормализация (NFKC).
2. **Лексиконы (denylist)** — поддерживаемые конфиг-списки: `profanity_ru`, `hate_slurs`, `contact_patterns` (regex: телефоны/`@`/ссылки), `banned_terms`. Списки лежат в конфиге/БД (НЕ хардкодом в коде и НЕ в этом документе), версионируются, правятся модератором без релиза.
3. **Классификатор** (ML/сервис) для контекстных случаев, которые denylist не ловит (скрытая ненависть, токсичность). Вывод модели — **untrusted**: пользовательский текст оборачиваем `<user_content>`, результат модели не исполняем (SECURITY T-15).
4. **Решение:**
   - hard-hit (мат/слур/контакты явные) → **блок ввода сразу**: API `content_blocked`, объявление не публикуется, UI показывает инлайн-ошибку.
   - вероятностный/контекстный → объявление в `pending_review` (FR-012), уходит в очередь модерации админки.
   - чисто → публикуется.
5. **Fail-closed:** при недоступности классификатора — не пропускаем «вероятностные», шлём в ручную очередь.

## 4. Что видит пользователь (UX, ≠ «тихо проглотили»)
- Инлайн-ошибка под полем: «Текст содержит недопустимые слова — отредактируйте, пожалуйста». **Не подсвечиваем сам мат/слур** (чтобы не обучать обходу) — общее сообщение.
- Для `pending_review`: «Объявление на проверке, обычно занимает до N минут».
- **Апелляция:** кнопка «Не согласен» → кейс в очередь модерации с пометкой «оспорено».

## 5. Бэкенд-контракт
- Эндпоинты публикации/чата/отзыва возвращают `error: content_blocked` (hard) или флаг `moderation_pending`.
- Каждое авто-решение и ручное действие → запись в `moderation_case` + `AUDIT_LOG` (кто/что/причина/версия лексикона).
- Метрики качества (PRD §7): accuracy ≥ 0.95, false-allow (пропуск мата/слура) ≤ 1%, p95 инференса < 1.5с.

## 6. Эволюция
- Лексиконы пополняются из очереди модерации (обратная связь).
- Поведенческие сигналы (повторные попытки протолкнуть запрещёнку) → в риск-скоринг пользователя (`ADMIN_BACKEND_TZ.md`, fraud).
